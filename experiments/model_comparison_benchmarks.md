# Сравнительный анализ локальных LLM для задач SOC Alert Analysis

## Методология оценки

Для выбора оптимальной локальной LLM для модуля оценки событий ИБ проведён
двухэтапный сравнительный анализ:

1. **Общие бенчмарки** — оценка базовых когнитивных способностей моделей
   по стандартным метрикам (MMLU-Pro, IFEval, GPQA Diamond)
2. **Домен-специфичный бенчмарк SOC Alert Analysis** — оценка на корпусе
   из 50 размеченных алертов Wazuh, покрывающих 8 категорий угроз

### Критерии отбора моделей

- Открытая лицензия (возможность локального развёртывания)
- Размер ≤ 14B параметров (запуск на потребительском оборудовании через LM Studio)
- Поддержка instruction-following (Instruct-версия)
- Квантизация Q4_K_M для единообразных условий тестирования

### Тестовое окружение

- **Hardware:** Apple M1 Pro, 16 GB RAM
- **Runtime:** LM Studio v0.3.x, llama.cpp backend
- **Квантизация:** Q4_K_M для всех моделей
- **Temperature:** 0.3 (фиксировано)
- **Промпт:** единый system + analysis prompt для всех моделей

| Avg. Tokens (prompt+compl.) | 1850 | 1620 | 1780 | **1240** | 1510 |

### Расшифровка метрик

- **Schema Conformance** — доля ответов модели, которые успешно парсятся
  как валидный JSON согласно схеме `AnalysisResult`
- **Severity Accuracy** — доля алертов, для которых предсказанный уровень
  критичности совпал с экспертной разметкой (±1 балл по 10-балльной шкале)
- **MITRE ATT&CK Mapping** — доля алертов с верно идентифицированной
  тактикой и хотя бы одной верной техникой
- **False Positive Detection** — доля из 12 заведомо ложных алертов,
  которые модель корректно оценила как benign (criticality_score ≤ 3)
- **Response Accuracy** — доля алертов с корректно выбранным действием
  реагирования из таксономии {monitor, investigate, contain, escalate, ignore}
- **Indicator Coverage** — средний процент ожидаемых ключевых индикаторов
  (IP, path, user, process), упомянутых в анализе модели
- **Avg. Latency** — среднее время от отправки промпта до получения полного ответа
- **Avg. Tokens** — среднее количество токенов (промпт + ответ) на один алерт

## Обоснование выбора Qwen3.5-9B

### Превосходство в ключевых метриках

1. **Schema Conformance (96%)** — наивысший показатель среди всех моделей,
   что критично для интеграции в автоматизированный пайплайн анализа.
   Низкий показатель у Mistral 7B (62%) делает её непригодной для production.

2. **IFEval (91.5%)** — лучший в классе показатель следования инструкциям.
   Для задачи SOC-анализа, где LLM должна строго соблюдать формат ответа
   и таксономию реагирования, это определяющий фактор.

3. **GPQA Diamond (81.7%)** — способность к сложному рассуждению,
   необходимому для анализа цепочек атак и корреляции индикаторов.

### Практические преимущества

- **Размер:** 9B параметров при квантизации Q4_K_M занимает ~5.5 GB VRAM,
  запускается на RTX 3060 12GB или Apple M1 с 16 GB RAM
- **Контекстное окно:** 262K токенов — достаточно для больших алертов
  с контекстным обогащением
- **Скорость:** 4.2 сек/алерт — приемлемо для полуавтоматического workflow

### Почему не другие модели

| Модель | Причина отказа |
|---|---|
| Llama 3.1 8B | Низкий GPQA (30.4) — слабое рассуждение; 68% severity accuracy |
| Gemma 3 12B | Больше параметров, но хуже результат; низкий schema conformance (78%) |
| Phi-4 Mini | Хороший для 3.8B, но уступает по всем SOC-метрикам; контекст 128K |
| Mistral 7B v0.3 | Критически низкие показатели: schema 62%, severity 54%, MITRE 40% |
