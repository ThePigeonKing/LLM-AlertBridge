## 2. Актуальность проекта

Центры мониторинга информационной безопасности (SOC) ежедневно обрабатывают тысячи событий, поступающих из SIEM-систем. Значительная доля этих событий составляет ложные срабатывания (false positives), что приводит к перегрузке аналитиков и замедлению реакции на реальные инциденты. По оценкам отрасли, от 40 до 70 % рабочего времени аналитика первой линии уходит на рутинный триаж событий — первичную классификацию, сбор контекста и принятие решения о необходимости дальнейшего расследования.

Одновременно развитие больших языковых моделей (LLM) открывает возможности для автоматизации начального этапа анализа инцидентов. Однако применение облачных LLM-сервисов в контексте SOC ограничено требованиями конфиденциальности: данные событий могут содержать чувствительную информацию об инфраструктуре организации (IP-адреса, имена хостов, учётные записи, индикаторы уязвимостей). Отправка таких данных на внешние серверы нарушает базовые принципы информационной безопасности.

В связи с этим возникает задача создания автономного модуля, который интегрируется с существующей SIEM-инфраструктурой и использует локально развёрнутую LLM-модель для семантического анализа событий без передачи данных за периметр контролируемой инфраструктуры.

## 4. Новизна и преимущества полученных решений

### 4.1. Архитектурное решение с локальной LLM

В отличие от существующих подходов, использующих облачные LLM-сервисы (OpenAI API, Google Gemini и др.), разработанный модуль принципиально основан на локальном выполнении языковой модели. Модель развёрнута на рабочей станции аналитика через LM Studio, а связь с облачным бэкендом осуществляется через зашифрованный VPN-туннель. Это исключает передачу чувствительных данных об инфраструктуре за пределы контролируемой среды.
В случае с реальной разверткой в рамках инфраструктуры компании модель LLM можно поднять на более мощном сервере, находящимся во внутреннем контуре компании, что точно так же обеспечит конфиденциальность данных, которые в неё попадают, а также позволит быстрее проводить анализ событий.

### 4.2. Структурированный формат анализа

Система формирует структурированный JSON-ответ, включающий шесть фиксированных полей: краткое описание события, гипотезу о природе инцидента, возможные причины, ключевые индикаторы, рекомендуемые проверки и оценку уровня уверенности модели. Такой формат упрощает интеграцию результатов анализа в рабочие процессы SOC и обеспечивает единообразие представления данных.

### 4.3. Нормализация и сохранение контекста

Реализован механизм нормализации сырых данных Wazuh в унифицированный формат с извлечением ключевых полей (идентификатор правила, описание, уровень критичности, информация об агенте, сетевые и пользовательские данные). Как исходные, так и нормализованные данные сохраняются в JSONB-столбцах PostgreSQL, что обеспечивает возможность повторного анализа и ретроспективного исследования.

### 4.4. Роль модели как ассистента

Языковая модель выступает исключительно как интеллектуальный помощник аналитика. Система не принимает автономных решений и не инициирует автоматического реагирования. Окончательное суждение всегда остаётся за человеком-оператором.

## 5. Степень обоснованности и достоверности полученных результатов

### 5.1. Обоснованность архитектурных решений

Выбор каждого технологического компонента документирован и обоснован:

- **Wazuh** — открытая SIEM-система с активным сообществом, богатой библиотекой правил и поддержкой MITRE ATT&CK.
- **FastAPI** — асинхронный Python-фреймворк с автогенерацией OpenAPI-документации и встроенной поддержкой Pydantic-моделей для строгой валидации данных.
- **PostgreSQL 16** — поддержка JSONB с индексацией, асинхронный доступ через asyncpg, версионируемая схема через Alembic.
- **OpenAI SDK с кастомным base_url** — стабильный интерфейс с встроенным механизмом повторных попыток и поддержкой потоковой передачи данных.
- **LM Studio** — OpenAI-совместимый API, возможность запуска на потребительском оборудовании (Apple Silicon / NVIDIA GPU).
- **Jinja2 + HTMX + Tailwind CSS** — серверный рендеринг с динамическим поведением UI без необходимости SPA-фреймворка.

### 5.2. Достоверность результатов бенчмаркинга

Сравнительный анализ моделей проведён на выборке из 50 размеченных событий Wazuh, покрывающих 8 категорий угроз (SSH brute force, file integrity, privilege escalation, web attacks, rootkit detection, audit/command execution, malware/C2, benign/false positive). Каждое событие размечено экспертом с указанием уровня критичности, действия реагирования, маппинга MITRE ATT&CK и ключевых индикаторов.

Тестирование проведено в единообразных условиях: все модели квантизированы в формате Q4_K_M, использовались идентичные промпты, фиксированная температура генерации (0.3), одинаковое аппаратное обеспечение (Apple M4 Pro, 24 GB RAM, LM Studio v0.3.x).

Тем не менее следует учитывать, что:

- размер тестовой выборки (50 событий) ограничен и может быть недостаточен для статистически значимых выводов по отдельным категориям;
- экспертная разметка выполнена ограниченным числом специалистов, что вносит субъективность;
- результаты справедливы для конкретной конфигурации квантизации и оборудования, но отражают общую степень производительности различных моделей LLM.

### 6.2. Архитектура системы

#### 6.2.1. Топология инфраструктуры

Система развёрнута в облачной инфраструктуре Yandex Cloud. Все серверные компоненты (FastAPI-бэкенд, PostgreSQL, стек Wazuh) размещены на узле `core-compute` (10.128.0.29). Два целевых хоста (`target-1-compute`, `target-2-compute`) с установленными агентами Wazuh выступают источниками событий. Узел `attacker-compute` используется для генерации тестовых инцидентов. Подключение аналитика к облачной сети обеспечивается через OpenVPN-сервер.

LLM-модель развёрнута на ноутбуке аналитика через LM Studio. Бэкенд обращается к ней по VPN-туннелю, используя OpenAI-совместимый API.

| Узел | Внутренний IP | Роль |
|------|---------------|------|
| `core-compute` | 10.128.0.29 | Backend + БД + стек Wazuh (Docker Compose) |
| `target-1-compute` | 10.128.0.35 | Агент Wazuh (мониторируемый хост) |
| `target-2-compute` | 10.128.0.14 | Агент Wazuh (мониторируемый хост) |
| `attacker-compute` | 10.128.0.36 | Симуляция атак |
| `openvpn-access-server` | 10.128.0.7 | VPN-шлюз |
| Ноутбук аналитика | VPN-адрес | LM Studio (порт 1234) |

![Сетевая топология проекта](<Project Arcitetrure-2.jpg>)

#### 6.2.2. Поток данных

1. Узел `attacker-compute` генерирует события безопасности на `target-1-compute` и `target-2-compute`.
2. Агенты Wazuh на целевых хостах обнаруживают события и передают их на `core-compute` (Wazuh Manager).
3. FastAPI-бэкенд выполняет запрос к Wazuh Indexer (OpenSearch) через локальное соединение.
4. События нормализуются и сохраняются в PostgreSQL.
5. По запросу аналитика контекст события отправляется в LM Studio на ноутбуке через VPN.
6. LLM возвращает структурированный JSON-анализ обратно в бэкенд через VPN.
7. Результат сохраняется в PostgreSQL и отображается в веб-интерфейсе.

### 6.3. Алгоритмы и программные решения

#### 6.3.1. Нормализация событий

Модуль `normalizer.py` преобразует сырое событие Wazuh в унифицированную структуру. Из вложенных JSON-объектов (блоки `rule`, `agent`, `data`) извлекаются ключевые поля: идентификатор и описание правила, уровень критичности, маппинг MITRE ATT&CK, данные агента, сетевые адреса, пользовательские данные, полный лог.

Числовой уровень Wazuh (0–15) переводится в текстовую шкалу критичности:

| Диапазон уровня Wazuh | Текстовая метка |
|------------------------|-----------------|
| 0–3 | info |
| 4–6 | low |
| 7–9 | medium |
| 10–12 | high |
| 13–15 | critical |

#### 6.3.2. Формирование промптов

Система использует два шаблона: системный (`system.txt`) и пользовательский (`analysis.txt`).

Системный промпт задаёт роль модели как эксперта SOC-аналитика, определяет формат JSON-ответа с шестью обязательными полями (`summary`, `hypothesis`, `possible_causes`, `key_indicators`, `recommended_checks`, `confidence_note`) и устанавливает ограничения: модель анализирует строго предоставленные данные, не изобретает фактов и не принимает автономных решений.

Пользовательский промпт содержит метаданные события (Rule ID, описание, уровень критичности, имя агента, временная метка) и сериализованные нормализованные данные.

#### 6.3.3. Парсинг ответа LLM

Модуль `llm_service.py` реализует многоуровневый парсинг ответа модели:

1. Попытка прямого разбора как JSON.
2. В случае неудачи — извлечение JSON из markdown-блока ` ```json ... ``` ` с помощью регулярного выражения.
3. При невозможности разбора — формирование минимального объекта `AnalysisResult` с сырым текстом в поле `summary` и предупреждением в `confidence_note`.

Валидация выполняется через Pydantic-модель `AnalysisResult`.

#### 6.3.4. Оркестрация анализа

Сервис `analysis_service.py` реализует конвейер анализа с механизмом повторных попыток (до 2 попыток):

1. Загрузка события из БД, установка статуса `ANALYZING`.
2. Формирование промпта из нормализованных данных.
3. Отправка запроса в LM Studio с замером времени выполнения.
4. Парсинг и валидация ответа.
5. Создание записи `Analysis` с сохранением результата, сырого ответа, имени модели, количества токенов и времени обработки.
6. Обновление статуса события на `COMPLETED` или `FAILED`.

#### 6.3.5. Дедупликация при импорте

При синхронизации событий из Wazuh каждое событие проверяется на наличие в БД по полю `wazuh_id`. Дубликаты пропускаются.

### 6.4. Формат выходных данных LLM

Каждый анализ события формирует структурированный JSON:


### 6.6. Безопасность и защищённость

- **Локальное выполнение LLM.** Весовые коэффициенты модели и процесс инференса полностью контролируются аналитиком на его рабочей станции. Данные событий не покидают контролируемую инфраструктуру.
- **Шифрование канала связи.** Передача данных между облачным бэкендом и ноутбуком аналитика осуществляется через зашифрованный VPN-туннель (OpenVPN).
- **Сетевая изоляция.** API бэкенда публикуется только на внутреннем IP-адресе `core-compute` (10.128.0.29), не привязывается к `0.0.0.0`. PostgreSQL не экспонируется за пределы хоста. Дополнительной защитой служат security groups Yandex Cloud.
- **Локальное взаимодействие с Wazuh.** Обращение к Wazuh API и Indexer происходит внутри `core-compute` (в контексте Docker-сети), без сетевого трафика наружу.
- **Отсутствие автономных действий.** LLM не имеет возможности выполнять действия в инфраструктуре: она предоставляет структурированный анализ, окончательное решение принимает человек.

## 9. Подробное описание разработанной системы

### 9.1. Цели и задачи системы

Система LLM-AlertBridge предназначена для автоматизации начального триажа событий SIEM с помощью локально развёрнутой языковой модели. Система решает задачу снижения когнитивной нагрузки на аналитика SOC за счёт предоставления структурированного первичного анализа каждого события.

### 9.2. Функционал

1. **Синхронизация событий.** Получение последних событий из Wazuh Indexer (OpenSearch) через REST API с дедупликацией по идентификатору.
2. **Нормализация.** Преобразование сырых данных Wazuh в унифицированный внутренний формат с извлечением ключевых полей и маппингом уровня критичности.
3. **LLM-анализ.** Формирование структурированного промпта, отправка в LM Studio, парсинг и валидация JSON-ответа с механизмом повторных попыток.
4. **Хранение.** Сохранение событий и результатов анализа в PostgreSQL с JSONB-столбцами и отслеживанием статуса обработки.
5. **Визуализация.** Веб-интерфейс с таблицей событий, детальной страницей события с отображением результатов LLM-анализа, кнопками синхронизации и запуска анализа.
6. **REST API.** Программный доступ ко всем функциям через JSON API с автоматически генерируемой OpenAPI-документацией.
7. **Health-check.** Эндпоинт проверки доступности сервиса и базы данных.
8. **Seed-скрипт.** Загрузка 8 образцовых событий для локальной разработки и демонстрации без подключения к Wazuh.

### 9.3. Веб-интерфейс

Веб-интерфейс реализован с использованием серверного рендеринга (Jinja2) с элементами динамического поведения (HTMX). Стилизация выполнена через Tailwind CSS (CDN). Интерфейс оформлен в тёмной цветовой схеме, соответствующей эстетике SOC-панелей.

Основные экраны:

1. **Список событий** (`/alerts`) — таблица с колонками: уровень критичности (цветовая маркировка), Rule ID, описание, агент, статус обработки, время. Кнопка «Sync from Wazuh» с индикатором загрузки (HTMX). Пагинация.
2. **Детальная страница события** (`/alerts/{id}`) — метаданные события, раскрываемые блоки с сырыми и нормализованными данными, панель LLM-анализа с кнопкой запуска (HTMX, подгрузка результата без перезагрузки страницы).

## 10. Экономическая эффективность

### 10.1. Значимость для области информационной безопасности

Проблема перегрузки аналитиков SOC является общеотраслевой. Согласно отраслевым исследованиям, аналитики первой линии SOC тратят значительную часть рабочего времени на первичную обработку событий, из которых существенная доля является ложными срабатываниями. Автоматизация начального триажа с помощью LLM потенциально позволяет:

- сократить время обработки отдельного события за счёт предоставления готового структурированного анализа;
- повысить долю выявленных ложных срабатываний без участия человека на начальном этапе;
- высвободить ресурсы аналитиков для расследования подтверждённых инцидентов.

### 10.2. Значимость для МИЭМ и НИУ ВШЭ

Проект находится на пересечении двух актуальных направлений — информационной безопасности и применения LLM — и может служить основой для дальнейших научных исследований, публикаций и учебных материалов.

### 10.3. Преимущества локального развёртывания

Использование локальной LLM вместо облачных сервисов исключает:

- операционные затраты на API-вызовы (стоимость токенов);
- риски утечки конфиденциальных данных;
- зависимость от доступности внешних сервисов.

При наличии оборудования с GPU (RTX 3060 / Apple M-серия) стоимость эксплуатации системы ограничивается стоимостью облачной инфраструктуры для Wazuh и электроэнергией.

## 14. Направления дальнейшего развития

По итогам проектной работы определены следующие направления возможного развития системы:

1. **Контекстное обогащение через osquery.** Интеграция с osquery для сбора телеметрии хостового уровня (процессы, сетевые соединения, пользователи), расширяющей контекст события для LLM-анализа. Предварительные результаты (раздел 8.6) показали прирост точности на 6–8 п.п.
2. **Диалоговый режим.** Возможность задавать уточняющие вопросы модели в контексте конкретного события с сохранением истории диалога.
3. **Поиск похожих инцидентов.** Реализация механизма поиска исторически похожих событий и инцидентов с ранжированием по степени сходства.
4. **Обратная связь от аналитика.** Механизм учёта оценок и классификаций аналитика для корректировки логики интерпретации.
5. **Расширенные эксперименты.** Многорежимное сравнение (без LLM / LLM / LLM + контекст), количественные метрики на большей выборке, оценка precision/recall/F1-score.
6. **Автоматизированное тестирование.** Разработка тестовых случаев для интеграционного и модульного тестирования.
7. **Асинхронная очередь задач.** Вынесение LLM-анализа в фоновую очередь (Redis/Celery) для обработки больших объёмов событий.
